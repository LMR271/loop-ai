# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

**loop-ai** — a Rails 8.1 app for collecting and analyzing feedback via "loops". A `User` owns `Loop`s; each loop has ordered `Question`s, collects `Feedback` (respondent transcripts + per-conversation sentiment), and produces an `Insight` (loop-level summary).

**Ownership is by workspace, not by `current_user`.** A `User` can either run their own workspace or accept an invite to join someone else's as a teammate, so almost every controller reads `current_workspace_owner` (the `User` whose data you should see — see `User#workspace_owner`) rather than `current_user`. Never scope loops/feedback to `current_user` directly. See the Teams/workspaces note under Architecture.

**The core loop is closed and working end-to-end as of 2026-07-16.** A user creates a loop, activates it (provisioning an ElevenLabs voice agent), shares `/i/:slug`, a respondent talks to the agent, and when the call ends ElevenLabs POSTs the transcript back — which the app records as a `Feedback` with a sentiment score and the AI's stated reasoning, rendered on the Analyze dashboard. This has been verified with real conversations, not just tests. **The loop-level `Insight` layer is now built and proven too** — a two-stage OpenAI pipeline that rolls transcripts into per-interview extractions and then a clustered loop-level summary (themes + feature requests with verbatim quotes). It was verified end-to-end against the **live** OpenAI API on 2026-07-21 (see "Feedback analysis" below), so this is no longer a frontier — it works.

Working parts today: `LoopsController` is a full CRUD (`index`/`new`/`create`/`edit`/`update`/`destroy`) plus `activate`/`deactivate`/`approve` member actions; `RespondentsController` (`show` + `signed_url`) is the public voice-interview flow (see below); `AnalyzeController` (`index` + `show` + `refresh`) renders a feedback-review dashboard (feedback list, charts, and the loop-level insight panel), selecting a loop by `slug`, with `refresh` triggering Stage-2 analysis; `ElevenLabsWebhooksController#create` ingests finished conversations as `Feedback`; `DashboardController` (`index` + `update_stat_preferences`) renders the home dashboard + stat tiles; `DeployController` (`index` + `send_invites`) is the share/distribute page; `TeamController` (`index`/`create`/`destroy`) and `InvitationsController` (`show`/`update`, public) run the teammate-invite flow. The `Questions`, `Feedbacks`, and `Insights` controllers are still empty class bodies with no routes — build them out as needed. A `loops.agent_id` (string) column holds the ElevenLabs agent ID, set on activation — it is the join key in **both** directions: outbound to start conversations (signed URL) and inbound to match a finished conversation back to its `Loop` (`Loop.find_by(agent_id: payload.data.agent_id)`). API keys come via `.env` (`dotenv-rails` present); `ELEVENLABS_API_KEY` and `ELEVENLABS_WEBHOOK_SECRET` are expected there.

**Loop lifecycle gates (approval + locking).** Two booleans/columns guard a loop beyond its `status` enum: `pending_approval` and `first_deployed_at`. A loop created or edited by a non-admin teammate is set `pending_approval: true` (admins' loops are auto-approved); `LoopsController#activate` refuses to provision a `pending_approval?` loop, and a workspace admin clears it via `#approve` (`POST /loops/:id/approve`). `first_deployed_at` is stamped the first time a loop is activated and makes `loop.locked?` true — a **locked loop can never be edited again** (`ensure_editable!` bounces `edit`/`update` to the deploy page), which is why deactivate/re-activate reuses the existing agent instead of re-reading a possibly-changed prompt. `destroy`/`activate`/`deactivate`/`approve` are all admin-only (`require_workspace_admin!`); `LoopsController#index` supports full-text search via `params[:q]` (see PgSearch below).

**AI / agent work**: the loop-respondent side targets ElevenLabs conversational agents, wired end-to-end for *provisioning* and the *respondent voice flow*:
- `app/services/system_prompt_builder.rb` (`SystemPromptBuilder.new(loop).call`) turns a `Loop`'s `description` (Goal) and `Question`s ordered by `position` (numbered list) into an ElevenLabs system-prompt string plus a fixed Rules block. String-only, no HTTP.
- `app/services/eleven_labs_agent_creator.rb` (`ElevenLabsAgentCreator.new(loop).call`) POSTs to the ElevenLabs `convai/agents/create` API (via the `rest-client` gem) using that prompt, and returns the new `agent_id`. It has a request timeout and wraps every failure (HTTP error, timeout, unreachable host, missing key, missing `agent_id`) in a single `ElevenLabsAgentCreator::Error` with a readable message. The request body has two siblings: `conversation_config` (what happens *during* the call — prompt, `llm: "qwen35-397b-a17b"`, `tts.voice_id`) and `platform_settings` (what happens *around* it). `platform_settings.data_collection.sentiment` declares a field for ElevenLabs' **separate** `analysis_llm` (defaults to `gpt-4o-mini`) to extract from the finished transcript; its `description` (built by the private `sentiment_description`) is the entire prompt that model gets, and interpolates `SENTIMENT_VALUES` so the vocabulary lives in one place. **Sentiment config is create-time only** — the create-once activation guard means existing agents never gain it; a loop must be freshly activated to produce sentiment.
- `LoopsController#activate` (`POST /loops/:id/activate`) is the trigger. Design decisions baked in: **provision on activation only** (not on create/edit — no surprise API calls), **create-once** (guarded — skips the call if the loop is already `active` or already has an `agent_id`, so no duplicate agents), and **synchronous** (calls inline and flashes success/error immediately rather than a background job). On success it sets `agent_id` + `status: :active`; on failure the loop stays `draft` with an error flash. The edit view shows an "Activate" button (draft) or an "Active" badge (active). `LoopsController#deactivate` (`POST /loops/:id/deactivate`) flips an active loop to `status: :closed` (keeping its `agent_id`, so re-activation reuses the existing agent). The activate/deactivate branching lives in private `activation_outcome`/`activate_loop!`/`deactivation_outcome` helpers. Full design in `docs/superpowers/specs/2026-07-14-activate-loop-provision-agent-design.md`.
- `RespondentsController` is the **public respondent voice flow** (`skip_before_action :authenticate_user!`). `GET /i/:slug` (`respondent_path`) looks the loop up by `slug` and renders `show` for `active?` loops or the `closed` view otherwise. `GET /i/:slug/signed_url` (`respondent_signed_url_path`) fetches a fresh ElevenLabs signed conversation URL for `@loop.agent_id` (guarded — `head :not_found` unless `active?`). The `interview_controller.js` Stimulus controller fetches that signed URL, then opens a browser voice session via the `@elevenlabs/client` importmap package, surfacing every failure in an `aria-live` status element.

**Transcript ingestion** — the inbound half, and the app's **only** inbound edge (everything else calls *out* to ElevenLabs; here they call *in*, with no `current_user` and no auth wall):
- `ElevenLabsWebhooksController#create` (`POST /webhooks/elevenlabs`, `eleven_labs_webhook_path`) — public (`skip_before_action :authenticate_user!` + `skip_forgery_protection`; ElevenLabs carries no CSRF token). Flow: verify HMAC → `head :unauthorized` if invalid (**the only non-200**) → ignore non-transcription events → `Loop.find_by(agent_id:)` → `Feedback.create!` → `LoopMailer.new_feedback(...).deliver_later` (notifies the loop owner) → `head :ok`. Reads `request.raw_post`, never `params`, because the HMAC covers the exact bytes. **Never log `request.raw_post`** — transcripts are respondent PII.
- `app/services/eleven_labs_signature_verifier.rb` — `ElevenLabsSignatureVerifier.new(header, raw_body).valid?`. HMAC-SHA256 over `"#{t}.#{raw_body}"`, constant-time compare, 30-minute staleness window against replay. Fails closed on a missing secret. There is no ElevenLabs Ruby SDK; their documented `construct_event` helper is Python/JS only, so this is hand-rolled.
- `app/services/eleven_labs_webhook_payload.rb` — wraps the raw body and owns **all** knowledge of ElevenLabs' undocumented shape (`transcription?`, `agent_id`, `conversation_id`, `transcript`, `sentiment`, `sentiment_rationale`). Prefers the hash form, falls back to the list form, degrades to `nil` on an unexpected shape rather than raising — a 500 here risks the workspace-wide webhook being disabled.
- `app/services/transcript_formatter.rb` — flattens the turn array into `Agent:`/`Respondent:` lines, mapping ElevenLabs' `user` role to `Respondent` to match `db/seeds.rb`. The Analyze view renders transcripts with `white-space: pre-wrap`, so real and seeded feedback look identical.
- Idempotency is guaranteed by the **unique index** on `feedbacks.conversation_id`, not by the `rescue ActiveRecord::RecordNotUnique` in the controller — an `exists?` check would race between simultaneous deliveries.

**Sentiment in the UI**: `AnalyzeHelper#sentiment_badge` renders a Bootstrap pill per `Feedback::SENTIMENT_VALUES`; `excited` gets a solid `text-bg-success` fill and the rest get `-subtle`/`-emphasis` pairings, so a long list stays calm while the signal you care about still pops. `nil` renders nothing (feedback from pre-`data_collection` agents was never scored, and an empty badge would imply it was). The `sentiment_rationale` renders beneath the transcript in muted italic — claim, then evidence, then reasoning, so a user can disagree with the AI rather than just trust it.

**Feedback analysis (the two-stage LLM pipeline — LIVE and proven)**. Built on branch `feature/lennart-feedback-analysis`, verified against the live OpenAI API on 2026-07-21. Turns raw transcripts into a structured, quote-backed loop-level `Insight`. Spec: `docs/superpowers/specs/2026-07-20-feedback-analysis-design.md`.
- **`app/services/llm_client.rb`** — the single OpenAI seam. `LlmClient.new.complete(system:, user:, schema:)` calls OpenAI Chat Completions (via the `ruby-openai` gem) with `MODEL = "gpt-5-mini"` and **structured outputs** (`response_format: {type: "json_schema", json_schema: {name:, schema:, strict: true}}`), returning a parsed Hash validated against `schema`. Wraps every failure in `LlmClient::Error`. Key comes from `ENV["OPENAI_API_KEY"]`. **Model id + request shape are confirmed correct against the live API — do not "fix" them.** All tests stub this class, so it is the *only* thing a smoke test actually exercises.
- **Stage 1 — `FeedbackAnalyzer.new(feedback).call`** (`AnalyzeFeedbackJob`): one transcript → writes `feedbacks.title`, `feedbacks.summary` (narrative), and `feedbacks.extracted_points` (jsonb `{"points" => [...]}`, each point `{kind: "theme"|"request", title, quote}` with the quote copied **verbatim**). Enqueued from `ElevenLabsWebhooksController#create` right after `Feedback.create!` (so real feedback self-analyzes on arrival), and by `rake analysis:backfill` for any feedback with empty `extracted_points`. **On `LlmClient::Error` it logs and writes nothing** — a broken key/quota is a silent no-op in the job path, not a raise. A smoke test must therefore assert on written data, not just "no crash."
- **Stage 2 — `AnalyzeLoopJob` → `LoopAnalyzer.new(loop).call` → `LoopInsightWriter`**: clusters the `extracted_points` of *all* analyzed feedbacks (those with non-empty `extracted_points`) into one JSON result, then persists it in a transaction — destroys the loop's old `insight` and rebuilds it. `LoopAnalyzer` returns `overall_sentiment` (∈ `Feedback::SENTIMENT_VALUES`), a narrative `summary`, `themes`, and `feature_requests`. `LoopInsightWriter` writes an `Insight` (`summary`, `overall_sentiment`, `analyzed_feedback_count`, `generated_at`) with child `themes` and `feature_requests`, each carrying `quotes`. **Hallucination guard**: it drops any citation whose `feedback_id` isn't actually in the loop, so the AI can't invent a source. Stage 2 is **on-demand**, triggered by `AnalyzeController#refresh` (`POST /analyze/:slug/refresh`, `refresh_analyze_path`) — it is not automatic.
- **Models**: `Loop has_one :insight`; `Insight` (unique index on `insights.loop_id` — one per loop) `has_many :themes, :feature_requests`; `Theme` (title/description/`mention_count`/sentiment) and `FeatureRequest` (title/description, `status` enum `open/planned/done/dismissed`, `github_issue_url`) each `has_many :quotes, as: :quotable`; `Quote belongs_to :quotable` (polymorphic) `+ :feedback` (verbatim `text`, back-linked to its source feedback). `Feedback` gained `title`/`summary`/`extracted_points` columns and `has_many :quotes`.
- **Staleness + notification bell**: `Loop#unanalyzed_feedback_count` = `feedbacks.size - insight.analyzed_feedback_count`. `ApplicationController` exposes `unanalyzed_feedback_total` / `loops_with_new_feedback` (helper_methods) driving the navbar bell badge (`app/views/shared/_navbar.html.erb`); the Analyze `_insight_panel` shows an "N new responses since" note when the count is positive.
- **Runtime prerequisites** (both required or the feature silently does nothing): `OPENAI_API_KEY` in local `.env` **and** Heroku config vars (jobs fail without it), and `SOLID_QUEUE_IN_PUMA=1` in production (no worker dyno — the Solid Queue supervisor runs inside the web dyno's Puma; without it the jobs enqueue and never run). Smoke test: `bin/rails runner` a script that calls `LlmClient` → `FeedbackAnalyzer` → `LoopAnalyzer`/`LoopInsightWriter` and asserts real data was written. Gotcha seen live: a `429` with `type: "insufficient_quota"` is an OpenAI **billing** state (no credits on the project), distinct from rate-limiting — the key still authenticates, so it's not a code bug.

### Verified ElevenLabs facts (confirmed against a real payload — do not re-derive)

ElevenLabs' docs specify **none** of the following; each was established empirically on 2026-07-16 and is pinned in `test/fixtures/files/elevenlabs_post_call_transcription.json` (a real 25KB `post_call_transcription` payload).

- **Signature**: header `ElevenLabs-Signature`, format `t=<unix_ts>,v0=<sha256_hex>`. Verified construction: `OpenSSL::HMAC.hexdigest("SHA256", secret, "#{t}.#{raw_body}")`. Hash the **raw body** — reparsing JSON changes the bytes and breaks the hash. Compare with `ActiveSupport::SecurityUtils.secure_compare`.
- **Sentiment**: `data.analysis.data_collection_results.sentiment.value` and `.rationale`. The identical data *also* arrives as `data.analysis.data_collection_results_list` (an array keyed by `data_collection_id`) — a free fallback. `data_collection_results` is `{}` for agents provisioned without `data_collection`.
- **Transcript**: `data.transcript` is an **array of turn objects** (`role`, `message`, `time_in_call_secs`), never flat text. Roles are `"agent"` and `"user"`; `db/seeds.rb` uses `Agent:`/`Respondent:` lines, so map `user` → `Respondent` to match. The respondent may speak first.
- **Free extras**: `data.analysis.transcript_summary` and `call_summary_title` are populated by default — the obvious feed for `insights.summary`.
- **Webhook scope**: registered **once, workspace-wide** (`GET/PATCH /v1/convai/settings`, `webhooks.post_call_webhook_id`), not per agent. Our code never registers it. Confirmed settings: `events: ["transcript"]`, `send_audio: false`.
- **Always return 200** except on a bad signature. ElevenLabs auto-disables a webhook after 10 consecutive failures (last success >7 days ago or never), which would kill ingestion for **every** loop. An unknown `agent_id` must be logged and 200'd, never 404'd. There are **no retries**: *"For HIPAA compliance, if a webhook fails we can not retry the webhook."*
- **`llm` values are an undocumented enum** whose spelling differs from the docs' display names — e.g. `qwen35-397b-a17b`, not `Qwen3.5-397B-A17B`. A wrong value 422s. The API's own validation error enumerates the valid list.

Not built yet (deferred, in rough order): **sentiment aggregation views** (a `sentiment` option in the Analyze `data_view` selector via `@loop.feedbacks.group(:sentiment).count`, and dedicated sentiment tiles in `DashboardStats` — the stat-tile framework itself is built, this just needs sentiment-specific keys); **`respondent_email` capture** (the column exists but the payload carries no email unless passed as a dynamic variable at session start — currently always nil, rendering as "Anonymous respondent" and feeding the `anonymous_feedback` stat); **audio** (`post_call_audio` event, currently disabled via `send_audio: false`); and **re-sync** (pushing question/`data_collection` edits to an already-provisioned agent — moot for a `locked?` loop, which can no longer be edited, but pre-`data_collection` agents can never produce sentiment).

## Commands

Ruby 3.3.9 (per `.ruby-version` and the `Gemfile`), Rails 8.1, PostgreSQL. Use the `bin/` wrappers.

**Bundler gotcha**: `Gemfile.lock` is `BUNDLED WITH 4.0.14`, but Ruby 3.3.9 ships Bundler 2.5.22. Every `bundle` command therefore tries to fetch and re-exec Bundler 4.0.14 first — if that fetch stalls, `bundle install` hangs with no output and looks like a network problem when it is a version-bootstrap problem. Install it up front (`gem install bundler -v 4.0.14`) rather than debugging the hang.

```bash
bin/setup                 # install deps, prepare DB, boot (--skip-server to skip boot)
bin/dev                   # run the dev server (alias for bin/rails server)
bin/rails db:seed:replant # wipe and reseed (db/seeds.rb: founder user + sample loops)

bin/rails test                              # run all tests (Minitest)
bin/rails test test/models/loop_test.rb     # single file
bin/rails test test/models/loop_test.rb:7   # single test by line number
bin/rails test:system                       # Capybara/Selenium system tests

bin/rubocop               # lint (rubocop-rails-omakase, see .rubocop.yml)
bin/brakeman              # security static analysis
bin/bundler-audit         # gem CVE audit
bin/ci                    # run the full CI pipeline locally (see config/ci.rb)
```

`bin/ci` is the source of truth for CI: setup → rubocop → bundler-audit → importmap audit → brakeman → `bin/rails test` → seed replant. Run it before considering work done.

**`bin/ci` is currently RED on `master`, and it is not your change.** `PagesControllerTest#test_signed-in_visitors_can_view_the_landing_page` expects a 200 but gets a 302 — commit `b0090cb` made `PagesController#home` redirect signed-in users to `dashboard_path` and the test was never updated. The redirect is intentional; the test is stale. Left as-is because it belongs to the dashboard feature, but it means a red suite is the baseline, so **check whether a failure is yours before chasing it** (and note a permanently red CI hides real breakage from teammates — worth fixing).

## Architecture notes

- **Auth**: Devise (`:database_authenticatable, :registerable, :recoverable, :rememberable, :validatable`). `ApplicationController` enforces `authenticate_user!` globally — each new controller is authenticated by default; use `skip_before_action :authenticate_user!` for public actions (as `PagesController#home`, `RespondentsController#show`/`#signed_url`, and `InvitationsController` do). Public actions that read a loop must guard on `active?` themselves (the respondent flow returns the `closed` view / `head :not_found` for non-active loops) — there is no auth wall to lean on.
- **Teams / workspaces**: the `Team` model is a **membership/invitation record**, not a group — it joins an `account_owner` (the founder `User`) to an invited teammate (`user`, nil until accepted) with a `role` enum (`admin`/`editor`) and a `has_secure_token :invitation_token`. `User#workspace_owner` returns the founder whose data the current user should see (themselves, unless they've accepted an invite); `workspace_role`/`workspace_admin?` derive their permissions. `ApplicationController` exposes `current_workspace_owner`, `current_user_workspace_admin?` (both `helper_method`s), and `require_workspace_admin!` — **scope every query through `current_workspace_owner`, never `current_user`**, or teammates see the wrong data. `TeamController` (admin-only) invites/revokes and emails via `TeamMailer.invite`; the invitee accepts through the public `InvitationsController` (`/invitations/:invitation_token`), which creates their `User`, stamps `invitation_accepted_at`, and signs them in. Existing-email invites are bounced to login (a teammate can't be created over an existing account).
- **Loop.slug** is generated by `has_secure_token :slug` (unique index) and is the lookup key for both the analyze dashboard (`/analyze/:slug`, auth-gated) and the public respondent flow (`/i/:slug`). `Loop.status` is `enum :status, { draft: 0, active: 1, closed: 2 }` (integer column, default 0); loops start `draft`, are flipped to `active` by `LoopsController#activate` once their voice agent is provisioned, and back to `closed` by `#deactivate`. `Loop` also declares `accepts_nested_attributes_for :questions, allow_destroy: true` with a `reject_if` that drops blank new questions, so loop forms manage their ordered `Question`s (ordered by `position`) inline. `questions_form_controller.js` (Stimulus) drives add/remove/reorder of question fields client-side, setting `position` and toggling `_destroy`.
- **Routes**: `config/routes.rb` has a single `resources :loops` block with `member { post :activate; post :deactivate; post :approve }`, the public respondent routes `get "i/:slug"` (`respondent_path`) and `get "i/:slug/signed_url"` (`respondent_signed_url_path`), the inbound webhook `post "webhooks/elevenlabs"` (`eleven_labs_webhook_path`), the analyze routes (`analyze_index_path` / `analyze_path`), the `dashboard` routes (`dashboard_path`, `dashboard_stat_preferences_path`), the `deploy` routes (`deploy_path`, `deploy_invites_path`), the `team` routes (`team_path`, `team_member_path`), and the public `invitation` routes (`invitation_path`). Note routes are hand-written `get/post` entries (not `resources`), so path helpers are custom — check the file rather than guessing.
- **Feedback schema**: `feedbacks` carries `loop_id`, `transcript` (text, flattened `Agent:`/`Respondent:` lines), `sentiment` (string, one of `Feedback::SENTIMENT_VALUES` — the constant lives on the model, not the service, so the prompt that *requests* a value and the validation that *accepts* one read the same list; `ElevenLabsAgentCreator` interpolates it into the prompt), `sentiment_rationale` (text — ElevenLabs' stated reasoning, so a sentiment claim can always show its evidence), `conversation_id` (string, **unique index** — idempotency plus the back-reference to ElevenLabs' conversation record, where the audio lives), and `respondent_email` (string, **currently always nil** — the column exists and `DashboardStats` counts `anonymous_feedback` from it, but nothing populates it yet; the payload carries no email unless passed as a dynamic variable at session start). `conversation_id` is nullable and Postgres permits unlimited `NULL`s in a unique index, so seeded feedbacks (which have none) coexist with real ones. Sentiment is deliberately **per-conversation**, not per-loop: `insights.sentiment` was removed (wrong grain, zero readers); `insights.summary` remains for the deferred loop-level rollup.
- **Local webhook testing**: ElevenLabs cannot reach `localhost`, so post-call payloads arrive through an ngrok tunnel (`ngrok http 3000`). `config/environments/development.rb` allow-lists `.ngrok-free.dev` / `.ngrok-free.app` / `.ngrok.io` in `config.hosts` — without this Rails returns a 403 "Blocked host" before the controller runs. Note `config/environments/*` is **not** hot-reloaded; restart the server after changing it. On ngrok's free tier the URL changes every restart, and an ElevenLabs webhook's URL **cannot be edited after creation** — a new tunnel means creating a new webhook. The respondent link (`respondent_url`) is generated from the current request host, so a local `127.0.0.1:3000` link is fine for testing (browsers treat localhost as a secure context, so the mic works).
- **Dashboard stats**: `app/services/dashboard_stats.rb` (`DashboardStats.new(loops, selected_keys).cards`) computes the stat tiles on the home dashboard from a loops scope. Which tiles show is **user-selectable and opt-in**: `users.dashboard_stat_keys` (jsonb array, default `[]`) holds the choice, capped at `MAX_SELECTED_KEYS` (4), falling back to `DEFAULT_KEYS`; `DashboardController#update_stat_preferences` persists it (intersected with `LABELS.keys` so only known keys survive). `DeployController#index` reuses the service for a single `active_loops` card. Add a new stat by extending `LABELS` + the private count loaders.
- **Deploy page**: `DeployController#index` lists draft vs active loops for sharing; `#send_invites` (`POST /deploy/:loop_id/invites`) parses a free-text blob of emails (split on whitespace/commas, `grep URI::MailTo::EMAIL_REGEXP`, dedup) and fires `LoopMailer.invite_respondent` per address. The activate/deactivate buttons on this page pass `return_to=deploy` so `LoopsController` redirects back here instead of the edit page.
- **Mailers**: `ApplicationMailer` defaults `from: "Loop AI Team <hi@getloop.me>"`. `LoopMailer` — `invite_respondent(loop, email)` (respondent share invite) and `new_feedback(feedback)` (owner alert, fired from the webhook on every ingested `Feedback`). `TeamMailer#invite(team_member)` sends the teammate invite. All are `deliver_later` (Solid Queue). Production delivery is **Postmark** (`config.action_mailer.delivery_method = :postmark`, `POSTMARK_API_TOKEN`); mailer URLs use `default_url_options host: "getloop.me"`.
- **Full-text search**: `Loop include PgSearch::Model` with `pg_search_scope :search_by_name_and_description` (weighted `name`/`description`, `tsearch` prefix matching), used by `LoopsController#index` when `params[:q]` is present. The `pg_search` gem is in the `Gemfile`.
- **Seeds**: `db/seeds.rb` is populated (a founder user + example loops/questions/feedback) and idempotent; `db:seed:replant` wipes and reloads it.
- **Frontend**: importmap-rails + Turbo + Stimulus (no Node build — npm packages are pinned into `config/importmap.rb`, e.g. `@elevenlabs/client` for the respondent voice session). Stimulus controllers live in `app/javascript/controllers/` (`interview` drives the voice flow, `questions_form` the nested-question editor, `clipboard`/`range_filter` support share links and analyze filters). Styling is Bootstrap 5.3 + Font Awesome via `sassc-rails`; SCSS is organized under `app/assets/stylesheets/{config,components,pages}` and imported through `application.scss`. Forms use `simple_form`. The palette lives in `config/_colors.scss` and is mapped onto Bootstrap's semantic names in `config/_bootstrap_variables.scss` (`$primary: $blue`, `$success: $green`, `$danger: $red`, `$warning: $orange`), so prefer Bootstrap utilities (including 5.3's `-subtle`/`-emphasis` pairings, which derive accessible tints automatically) over bespoke component CSS. `config/_design_tokens.scss` **is** imported (`application.scss:3`, before `config/colors`) and is the live source of the `$color-*`/`$radius-*`/`$space-*`/`$shadow-*` tokens and the `--color-*` custom properties, several pages (`_loops.scss`, `_deploy.scss`, `_analysis.scss`) already depend on it directly. `config/_colors.scss` only remaps a handful of Bootstrap-facing names (`$blue`, `$red`, etc.) onto those same `$color-*` values — it does not shadow or replace them.
- **Background/infra**: Solid Queue (jobs), Solid Cache, Solid Cable — all database-backed, no Redis. Relevant when adding async work (e.g. generating insights from feedback). Separate schemas: `db/queue_schema.rb`, `db/cache_schema.rb`, `db/cable_schema.rb`.
- **Deploy**: **live on Heroku** at `https://loop-ai-66a0d89256a1.herokuapp.com` (app `loop-ai`, EU, basic dyno) as of 2026-07-17. Deploy with `git push heroku <branch>:master` — Heroku only builds its `master`. The `Procfile` declares `release: bin/rails db:migrate` (runs before new dynos take traffic; a non-zero exit aborts the release and the old version keeps serving) and a `web` process. **No worker dyno** — `app/jobs/` now holds `AnalyzeFeedbackJob` and `AnalyzeLoopJob` (the feedback-analysis feature's Stage 1/Stage 2 LLM jobs) alongside `ApplicationJob`, so Solid Queue has real work to run; `config/puma.rb:38` runs the supervisor inside the web dyno if `SOLID_QUEUE_IN_PUMA` is set, avoiding a second dyno — **that env var must be set in production or these jobs enqueue and never run**. Kamal (`config/deploy.yml`, `.kamal/`) is still configured but unused.
  - **Single database**: Heroku Postgres provides one `DATABASE_URL`, so `production:` in `database.yml` is a single entry and Solid Cache/Queue/Cable all share the primary connection — there is no `connects_to` in `production.rb`/`cable.yml` and no `database:` in `cache.yml`. Their tables (13 of them) live in `db/migrate` alongside the app's own; `db/{cache,queue,cable}_schema.rb` are **deleted**, since schema files are only loaded per-database by `db:prepare` and never by `db:migrate`. Do not reintroduce a named `cache`/`queue`/`cable` database entry — Rails would treat each as a separate database for schema loading again.
  - **Config vars**: `RAILS_MASTER_KEY` and `ELEVENLABS_API_KEY` are set. `POSTMARK_API_TOKEN` is required for production email (invites, teammate invites, new-feedback alerts) — mail silently fails without it. `OPENAI_API_KEY` is required for `LlmClient` (the feedback-analysis LLM calls made by `AnalyzeFeedbackJob`/`AnalyzeLoopJob`) — those jobs fail without it. `ELEVENLABS_WEBHOOK_SECRET` is **not** set — transcript ingestion is dead in production until an ElevenLabs webhook is created for the production URL. The webhook is workspace-wide and its URL cannot be edited after creation, so pointing it at Heroku ends local ngrok testing until it is pointed back.
  - **HTTPS + mail domain**: `production.rb` now sets `config.force_ssl = true` (all traffic redirected to HTTPS; matters for the respondent mic, a secure-context feature). Mailer links use `default_url_options host: "getloop.me"` (the intended custom domain) rather than the raw `herokuapp.com` dyno URL — so generated invite/analyze links point at `getloop.me`; keep this in sync with the actual DNS.
  - **Fresh-database gotcha (fixed, worth knowing)**: `rename_table` in Postgres does not rename the table's sequence, and the schema dumper then pins the old sequence name as an explicit `default:` on the id column. That schema works on the machine where the migration ran and fails on every database built fresh from it. Cost a red deploy on 2026-07-17 (`teams` still pointing at `memberships_id_seq`); fixed by `20260717100300_rename_memberships_sequence_to_teams`. After any future `rename_table`, rename the sequence too and check `grep nextval db/schema.rb` is empty.

## Conventions

`.rubocop.yml` is a **custom** config (not rubocop-rails-omakase, despite `bin/rubocop` pulling omakase in) with `NewCops: enable`. It excludes `test/`, `config/`, `db/`, `bin/`, and a few others; disables `Style/StringLiterals`, `Style/FrozenStringLiteralComment`, `Metrics/AbcSize`, `Metrics/CyclomaticComplexity`, and several other cops; and sets line length to 120. Note `Metrics/MethodLength` (max 10) and `Metrics/ClassLength` (max 100) **are** on — keep methods short. The repo is not currently offense-free (`app/controllers/analyze_controller.rb` has pre-existing offenses); avoid adding new ones. Match surrounding style; run `bin/rubocop` before committing.

**Testing gotcha**: the bundled minitest (6.x) ships no `Mock`/`stub`, and no mocking gem is installed. `test/test_helper.rb` provides a small `stub_instance_method(klass, name, replacement) { ... }` helper (swap-and-restore) for stubbing at a seam — use it (e.g. stub `ElevenLabsAgentCreator#call` or its private `#post`) rather than reaching for `.stub`.
